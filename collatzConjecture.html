Collatz Conjecture Program in Java

Overflow

The main reason the Collatz Conjecture is so difficult is because of the infinite possibilities of integers. In theory, the program I created could be put on a loop to test every possible integer. The scale of these numbers is too large for any programing language to compute.

Try-Catch

To prevent the program from crashing when an integer reached the 32 bit limit of Java, I implemented a "try-catch" algorithm. This way, the program can run as efficiently as possible while staying in the scope of possible and impossible calculations.

Goals and Proceedure

The Collatz Conjecture is one of the simplest and most puzzling unsolved mathematical problems. In short, the idea states that if an integer is positive to divide it by two. If an integer is negative, multiply it by three and add one. The resulting integers are hailstone numbers, and all hailstone numbers should lead to an infinite loop starting with four, then to two, then to one. Almost all integers up to the billions have been tested and all end up in the "4-2-1" loop.

This project was not aimed to find a solution to the conjecture, but to map out hailstone numbers for certain starting integers. The mind-boggling fact behind the Collatz Conjecture is that there is no truly defined pattern between an integer and its hailstone numbers. As seen almost everywhere in nature, patterns tend to connect in multiple areas. If a pattern can be found, not only can the conjecture be proven true or false, but patterns for encryption and other situations can be developed. This uses three efficient algorithms to find the hailstone numbers, find the largest hailstone number, and display the hailstone tree on a simple user interface.

The main problem I ran into when programing was involving the gigantic values that some hailstone values reached. If a value was too large, it wouldn't compute because it would be out of the range of Java's 32 bit integer limit. In response, I created a try-catch algorithm that displayed an error message on the interface if this limit was reached. This avoided Java from running indefinitely and the program ultimately crashing.
